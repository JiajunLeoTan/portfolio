// 01 · data scale
What's in the Database

Three Amazon datasets ingested through a Python/pandas/psycopg2 ETL pipeline — product metadata, a review/comment export, and the McAuley Lab Pure-IDs 5-Core dense interaction dataset. Entity resolution deduplicates products by ASIN, canonicalizes brand names from noisy product titles (5-product minimum threshold), falls back from child ASINs to parent ASINs for reviews, and removes orphan rows before ingestion.

schema design
5-table BCNF schema: Categories (248) → Products (1.4M) ← Reviews (3M) → Users (1.28M), with Brands as a separate entity extracted from product titles. No display names stored redundantly on Products — category and brand names join in at query time. Three materialized views (mv_category_compare, mv_brand_performance, mv_value_score_components) serve as derived read models without breaking normalization.

// 04 · performance
Query Optimization Results

Benchmarked with EXPLAIN ANALYZE on a live AWS RDS instance (5 measured runs, 1 discarded warmup, 1 s pause between phases). The three most impactful wins all come from the same root cause: expensive cross-table aggregations running live on every request, eliminated by precomputed materialized views with covering indexes.

// matviews
Materialized Views
mv_category_compare — precomputed category price/rating aggregates. mv_brand_performance — brand product and review rollups. mv_value_score_components — normalized per-product score dimensions with an indexed default Balanced score (0.4/0.2/0.2/0.2).
// indexing
Composite Indexes
Trigram index on product titles for keyword search. Partial expression index on (1 - price/list_price) for valid discounted products (keeps Deals sub-millisecond). Review timestamp + ASIN index for time-window filtering. Covering index idx_mv_value_components_default_score_cover for the precomputed value score path.
// query restructuring
Query Rewrites
Helpful-reviews: rank top 10 reviews first, then compute reviewer stats only for those users (eliminates full-table reviewer aggregation). Category/brand listing: page product rows before joining display names. Value rankings: split default vs custom weight paths so the common Balanced preset uses the precomputed indexed score rather than re-scoring.
// caching
Route-Level TTL Cache
In-memory TTL cache for category/brand metadata, category comparison, and review trend by category. Review trend SQL is already selective enough that the first EXPLAIN ANALYZE is comparable — the cache targets repeat users where latency matters most.
top-value anomaly
The top-value query baseline was 322,838 ms — over 5 minutes — because it used correlated subqueries to compute per-product category averages, re-scanning the full 1.4M products table once per row. The optimized path reads precomputed cat_avg_price and cat_avg_stars directly from mv_value_score_components — same output, 652× faster at 494 ms.
