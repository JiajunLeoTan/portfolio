Autonomous Vehicle
Lane Change with MPC

A full-stack autonomous driving framework — from noisy perception through behavior planning to model predictive control — enabling an ego vehicle to detect surrounding traffic, decide when to change lanes, generate a dynamically consistent reference, and track it using a Drake-backed MPC over a 6-state bicycle model.

Vehicle Model 6-state dynamic bicycle
Controller Receding horizon MPC (Drake)
Solver SNOPT
Success rate 29/30 (straight traffic)
// 01 · system design
5-Layer Architecture

Each layer has a single responsibility and a clean interface to the next. Perception feeds noisy obstacle estimates up to the behavior planner; the planner outputs a 4-state reference request; trajectory generation expands this to a 6-state horizon; the MPC solves for optimal controls; the bicycle model integrates forward. Any layer can be swapped independently.

Perception
NoisyPerception
5 Hz low-rate obstacle sensing with Gaussian position noise (σx=0.5 m, σy=0.3 m). Simulates realistic sensor uncertainty — the behavior planner never sees ground-truth obstacle positions.
5 Hz updateGaussian noiseobstacle list
Behavior
ObstacleAwarePlanner (extends BehaviorPlanner)
Finite-state machine (CRUISE / LANE_CHANGE_TO). Scores all lanes by distance + TTC to nearest lead vehicle, with a center-lane bias. Hysteresis requires 3 consecutive frames of agreement before committing to a lane change. Post-change cooldown prevents oscillation. Outputs a TrajectoryRequest with target lane, cruise speed, and pre-computed quintic coefficients.
FSMTTC scoringhysteresiscooldownquintic splines
Trajectory
ReferenceProvider + TrajectoryPlanner
Converts the behavior request into a sampled 6-state reference horizon (x, y, ψ, vx, vy, ω) over T=15 steps. CRUISE mode: lane-centered straight path. LANE_CHANGE mode: executes the pre-stored quintic polynomial for lateral displacement, computing heading from dy/dx to align the vehicle with the path.
quintic polynomialC² continuous4 s maneuverheading from dy/dx
MPC
DrakeMPC (SNOPT solver)
Receding-horizon optimization over T=15 steps (Δt=0.1 s). Decision variables: states x and controls u. Linearizes bicycle dynamics about the reference trajectory at each step (numerical Jacobian via central differences). Quadratic cost penalizes tracking error, control effort, and control rate. Only the first control is applied.

Dynamics
VehicleBicycleModel
Rear-wheel-drive, front-wheel-steering bicycle model. State: [X, Y, ψ, vx, vy, ω]. Inputs: [δ, ax]. Lateral tire forces from small-angle slip approximation (Cf=Cr=80,000 N/rad). Euler-discretized for MPC; continuous model used for ground-truth simulation. Parameters match a lightweight sports car (~Mazda Miata).

// 02 · contributions
What I Built
// perception
NoisyPerception Module
Designed and implemented the perception layer — a low-rate (5 Hz) sensor model with independent Gaussian noise on obstacle X and Y positions. The module updates on a configurable timer, decoupling sensor latency from the simulation loop and exposing the behavior planner to realistic uncertainty rather than ground truth.
// planning
Behavior Planner
Authored the full behavior planning stack — BehaviorPlanner base FSM and the ObstacleAwarePlanner extension. Designed the TTC + distance lane-scoring heuristic, the hysteresis/cooldown anti-oscillation logic, the quintic polynomial maneuver generator, and the intersection conflict detection mode.
// control
Initial MPC Controller
Developed a prior version of the MPC controller before the final linearized Drake implementation. This included the nonlinear Euler-step dynamics formulation, constraint setup, and the cost function structure — providing the foundation that the final linearized version was built on top of.
// 03 · test scenarios
5 Evaluation Scenarios

The framework was evaluated across five progressively harder scenarios. It performs well in its design envelope — straight, unidirectional traffic — and the failure modes outside that envelope are well-understood and point to concrete future work.

Multi-Obstacle Overtake 29/30
1–6 randomly placed obstacles with noisy 5 Hz perception. Ego vehicle smoothly overtakes slower traffic. Single failure at 4 obstacles due to perception noise reducing safety margin.
Impassable Traffic Wall partial
All lanes blocked by vehicles at equal speed. Ego successfully matched lead vehicle speed. Perception noise caused oscillatory lane-change attempts — a structural planner issue, not fixable with better sensors alone.
Highway Merge partial
Merging into fast-moving traffic with a slow vehicle ahead. MPC input limits correctly prevented unsafe inputs. Failed when a stalled vehicle required simultaneous large acceleration + steering — exposed lack of a "yield and wait" mode.
Curved Road failed
Quarter-circle arc (R=100 m) at 18 m/s. Vehicle drifted inward — the MPC was linearized about zero steering, penalizing the nonzero vy and ω physically required to hold a curve. Curvature-aware feedforward steering needed.
4-Way Intersection failed
Cross-traffic from multiple directions. Behavior planner issued abrupt lane-change commands that the MPC could not execute feasibly, leading to collision. A dedicated intersection deceleration/yield mode is required.

known limitation
The root failure mode across most hard scenarios is the same: the behavior planner can issue a reference that changes too abruptly for the MPC to track dynamically. Past a certain yaw velocity, the bicycle model simply doesn't have enough control authority to execute the planned maneuver. Hysteresis and cooldown timers help but don't fully resolve this — a more anticipatory planner or a unified planning-control formulation is the right long-term fix.
