ACLN Search Engine

A cloud-deployable mini web search engine built end-to-end on a distributed KVS + Flame stack — robots-aware crawler, iterative PageRank, TF-IDF inverted index, and a custom HTTP server with ranked search, PDF ingestion, spellcheck, search suggestions, and 9-category query intent detection. Deployed on Amazon EC2, crawling 100–150K pages in under 9 hours.

// 01 · system design
End-to-End Pipeline

Four components, each running as an independent process or Flame job, connected entirely through KVS tables — no direct inter-component communication. The clean separation means batch jobs (crawler, PageRank, indexer) can be re-run or restarted independently without corrupting the others, and the webserver issues only KVS lookups at query time with no batch computation in the serving path.

0
KVS + Flame kvs/ · flame/
Course-provided distributed substrate. KVS workers shard tables by consistent key partitioning with durable disk storage. Flame provides an RDD-style MapReduce API — jobs are submitted as JARs over HTTP to a Coordinator, distributed to Workers as tasks. All persistent project data lives in KVS; no local files between components.

1
Crawler Crawler.java// leo
Runs as a Flame job seeded from one or more URLs. Fetches pages via HTTP, respects robots.txt per host, and enforces per-domain politeness delays. Extracts and normalizes outbound links with UrlUtils, deduplicates content by hash in contentSeen, and stores page HTML, metadata, and a semicolon-separated outbound link list in pt-crawl. Supports configurable page limits, blacklist filtering, multiple seed URLs, and verbose progress logging. Tunable request timeout and per-host scheduling drove the 9-hour final crawl throughput.

2
PageRank PageRank.java// leo
Iterative Flame job over the link graph stored in pt-crawl. Consumes cleaned outbound-link data rather than re-parsing HTML, using hashed URLs as keys throughout. Explicitly emits (targetHash, "") for target-only nodes so all reachable pages receive PageRank mass. Handles dangling nodes with explicit mass redistribution: base = (1−D)/N + D·danglingMass/N. Convergence checked by joining old and new rank RDDs and computing max |new − old|. Intermediate RDDs aggressively destroyed each iteration to prevent KVS disk accumulation. Final scores streamed directly to pt-pageranks via KVSClient.put().

3
Indexer Indexer.java// leo
Flame job that consumes pt-crawl, strips HTML with jsoup, enforces Latin-only token constraints, optionally applies Porter stemming, and optionally stores word positions for phrase queries. Produces three output tables: pt-index (token → posting list), pt-termfreq (token → docHash:tf per document), and pt-docstats (docHash → URL + token count). Column names hashed via Hasher.hash(rowKey + "|" + value) to avoid collisions. All outputs streamed directly to KVS; large intermediates destroyed eagerly to minimize disk footprint.

Web Server + Search webserver/ · search.html// leo
Custom HTTP server — thread pool, routing, sessions — entirely in-house to remain within course constraints. SearchHandler tokenizes queries, retrieves postings from pt-termfreq, document lengths from pt-docstats, and PageRank from pt-pageranks in parallel KVS lookups, computes normalized TF-IDF and PageRank, and blends them with URL/title boosts. Additional features: spellcheck (edit distance ≤2, 20K-entry scan), prefix-based search suggestions, 9-category query intent detection, and infinite scroll. PdfUploadHandler integrates Apache Tika to extract PDF text and inject it into pt-crawl with a synthetic upload:// URL, making uploaded documents immediately searchable without re-running the batch indexer.

KVS table schema
pt-crawl
Per-URL rows. Columns: url, page (HTML), responseCode, contentType, outbound links (semicolon-delimited hashes), canonicalURL. Primary crawl corpus.
pt-pageranks
Keyed by Hasher.hash(url). Column: rank (double). Output of the iterative PageRank Flame job.
pt-index
Inverted index. Row key: token. Value: CSV of url entries (base) or url:positions (positional mode).
pt-termfreq
Row key: token. Columns: docHash:tf pairs. Used by SearchHandler for per-document TF computation.
pt-docstats
Row key: docHash. Value: url\x01length (tab-separated URL and token count). Provides document length for TF-IDF normalization.
hosts / contentSeen
Crawler support tables. hosts: per-domain robots rules and last-access time. contentSeen: content hash deduplication across crawl workers.
// 02 · ranking
Scoring Formula

Two primary signals — TF-IDF content relevance and global PageRank importance — blended linearly, with small bounded bonuses when query terms appear in the URL or extracted title. Both signals are normalized per result set before blending, so neither dominates by scale.

ranking formula
score(url, query) = α · TFIDFnorm(url, query) + β · PageRanknorm(url)
— where α = 0.7, β = 0.3 —

tf = 1 + log(raw term count)    // per query term, per document
idf = log((N + 1) / (df + 1)) + 1    // N = total docs, df = docs containing term
TF-IDF = Σ(tf · idf) / doc_length    // summed over query terms, normalized
PageRank = pt-pageranks[hash(url)] / max(ranks in result set)
query intent detection — 9 categories

Each query is classified into the highest-scoring intent category based on keyword matching and structural signals (price patterns, citation markers, date patterns, URL-like queries). The detected intent surfaces task-specific result cards above the ranked list — e.g., a weather widget, shopping links, or dictionary-style definitions.

// 03 · crawl performance
Three Iterations, 9 Hours Final

Each run informed the next. The first run exposed per-host delay tuning opportunities and visibility gaps in domain coverage. The second validated a refined configuration on a Wikipedia local test before scaling. The final EC2 run brought the system to production readiness with a stable coordinator, improved timeout handling, and more effective idle-worker utilization — all without sacrificing robots compliance.

key design insight
Standardizing on hashed URLs as primary keys across all KVS tables was the single decision that simplified the most integration work. Once the crawler, indexer, and PageRank job agreed on Hasher.hash(url) as the canonical key format, the search frontend reduced almost entirely to orchestrating KVS lookups and score aggregation — no parsing, no re-fetching, no inter-component coordination.
