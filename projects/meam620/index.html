// 01 · overview
What Makes This Hard

Projects 1 and 2 solved halves of autonomous flight in isolation: planning and control with ground-truth state, and state estimation without full autonomy. Project 3 closes the loop — the controller now receives VIO estimates instead of truth, and every component interacts. A trajectory that is feasible under perfect feedback can fail under estimation drift, and aggressive accelerations worsen the very estimation errors they depend on.

The key design choice throughout was to prioritize stability over speed: conservative controller gains, iterative time-stretching on generated trajectories, and a 1.35× time scale factor for the EC local planner. This sacrifices raw competition time in exchange for the estimator staying coherent through the full flight.

// 02 · system architecture
Full Autonomy Stack

Eight steps, running in a closed loop at sensor rate. The pipeline is the same in both normal and EC modes — the EC version adds a replanning check and local goal selection between steps 2 and 6.

1
VIO State Estimation vio.py
Error-State Kalman Filter with a 18-dimensional error state. IMU propagation: integrates measured acceleration and angular velocity to update position, velocity, and quaternion attitude, while propagating the 18×18 error covariance via the linearized process Jacobian Fx with additive noise Qi. Visual updates: each observed feature landmark generates a 2D projection innovation; outliers above the threshold are rejected; Kalman gain K applies a Joseph-form update to the covariance and corrects all 18 error states including accelerometer bias, gyro bias, and gravity vector.

2
Occupancy Map Update occupancy_map.py
Voxel grid inflated by the collision margin. In EC mode, the map is rebuilt around the current estimated position at each replan — only the local 3.5 m region is considered free space, forcing the planner to discover the environment incrementally.
0.15 m voxelsinflation marginEC: local update
3
A* Graph Search graph_search.py
26-connected search on the voxel grid with metric Euclidean edge costs. Heuristic uses the exact goal position rather than the goal voxel center for a tighter admissible estimate and better tie-breaking. Tries four decreasing collision margins (0.40→0.25 m) before falling back to a straight-line path. Deadline-abort mechanism prevents search from blocking the simulator on large worlds.

4
Waypoint Pruning + Local Goal Selection world_traj.py
Greedy line-of-sight pruning removes redundant waypoints while respecting clearance. In EC mode, a local goal is picked at the furthest reachable point along the straight line to the global goal within the sensor horizon — switching to the global goal directly once it falls within range.
greedy pruningLOS clearanceEC: local goal3.5 m horizon
5
Minimum-Jerk Trajectory Generation world_traj.py
Piecewise quintic polynomials through pruned waypoints. Segment durations are initialized from distance-based rules, increased proportionally at turns, and iteratively stretched whenever sampled velocity, horizontal acceleration, thrust, or vertical acceleration exceed conservative limits. Scenario-adaptive speed profiles (vertical, long horizontal, switchback) tune the limits based on path geometry. In EC mode, each local segment is shifted into absolute simulator time before installation and includes a hover tail so the trajectory remains valid during replanning.
quintic splinemin-jerkiterative time stretchscenario profilesEC: absolute timehover tail
6
SE(3) Geometric Controller se3_control.py
Outer loop (position PD): computes desired acceleration from position/velocity errors with feed-forward, then projects the desired force onto the current body z-axis for thrust. Inner loop (attitude PD on Euler angles): tracks roll, pitch, and yaw errors with feed-forward yaw rate. Motor speeds computed through the 4×4 thrust/drag mixing matrix with rotor-speed saturation. Gains tuned to be strong enough for goal completion but not so aggressive that VIO errors amplify into unstable tracking.

Autograder Results — 6 Maps

The standard planner passed all 6 maps cleanly. The EC local replanner passed 3 maps safely — the same 3 where the full-world planner produced relatively short, open paths. Slalom reached the goal but the true flight path grazed an obstacle. Stairwell and switchback timed out — conservative 1.35× timing adds overhead that accumulates over long, constrained routes.

EC failure analysis
Slalom collided because straight-line local goal selection chose subgoals misaligned with the narrow corridor — the planner reached the goal but the true flight path cut a corner. Stairwell and switchback timed out because repeated A* calls and 1.35× timing compound over long constrained paths. The fix is corridor-aware local goal selection (frontier-based or via a visibility graph) rather than simple projection toward the global goal.

The normal planner reaches the goal in 6.5 s with tracking error under 0.2 m throughout. The EC planner takes 23.8 s and shows the characteristic step-like distance-to-goal convergence of repeated short-horizon replanning — each segment completes, the quadrotor briefly holds position, and the next local target brings it closer. Tracking error under EC is higher (up to ~0.35 m) but stays bounded, confirming the 1.35× timing was conservative enough to keep VIO stable.
// 04 · lessons learned
Key Engineering Insights

Estimation, planning, and control are coupled. A trajectory that looks geometrically valid can still fail because aggressive acceleration worsens VIO drift, which worsens tracking error, which causes the quadrotor to clip obstacles the planner had cleared. The right level of aggression depends on the estimator's noise tolerance, not just the dynamics.

Absolute time is non-negotiable for local replanning. The simulator queries the trajectory with absolute simulation time. Each new local segment is naturally generated in relative time. Forgetting to shift segments caused the simulator to interpret local subgoal endpoints as the mission endpoint and terminate early — a silent correctness bug that looked like a success.

State continuity across replan boundaries is unsolved. Replanning from a nonzero-velocity state can produce zig-zagging if the new polynomial is initialized inconsistently. Passing current velocity into the new segment and enforcing a minimum first-segment time helps, but the right solution is to blend segments before the active one ends, preserving velocity continuity at the seam.
